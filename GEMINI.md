# カスタム指示

## 1. 返答スタイル

- 挨拶・前置き・絵文字は禁止。
- 常に結論ファーストで答える。
- 不要な説明を避け、実装・判断・確認結果を簡潔に伝える。
- 作業中の進捗報告は、必要な場合のみ短く行う。
- 曖昧な点がある場合は、推測で進めず、何が不明かを明示して質問する。

## 2. ツール・実行方針

- 適宜、必要な Agent Skills を参照する。
- 実装・レビュー前に、全体検索へ入る前の参照先として `.agents/file-map.md` を確認する。
- `.agents/file-map.md` で該当する作業領域を特定し、まずそこに列挙されたファイルと直接 import 先、変更ファイル、検証対象だけを読む。
- ユーザーが明示しない限り、`dist`, `node_modules`, `test-results`, `*.log` は読まない。ログ確認が必要な場合も末尾またはエラー周辺に限定する。
- 長時間実行する処理、開発サーバー、監視系コマンド、並行確認は `tmux` セッションで実行する。
- `tmux` は作業継続・ログ保持・並行実行のために使う。危険操作を無条件に通す目的では使わない。
- 承認不要にしてよいのは、読み取り・検索・差分確認・ビルド・テスト・開発サーバー起動など、破壊的でないコマンドに限定する。
- `rm`, `git reset`, `git clean`, `git checkout --`, 広範囲の `mv`, 依存追加、外部スクリプト実行、デプロイ、本番環境操作は必ず事前確認する。
- ファイル編集は、可能な限り差分が明確な方法で行う。
- 実装後は、可能な限り実際の動作確認まで行う。
- UI実装後は、可能な限りモバイル幅のブラウザ表示で主要フローを操作して検証する。

## 3. 設計方針

- 長期的なコードのメンテナンス性とスケーラビリティを考慮する。
- 既存コードの設計・命名・スタイルに合わせる。
- 必要以上に抽象化しない。
- 単発用途のための汎用化・設定化・拡張性追加は避ける。
- 変更はユーザーの依頼に直接関係する範囲に限定する。

## 4. モバイルUI/UXの必須要件

モバイルブラウザでの片手操作を前提に、全画面・全操作で「直感的な操作」と「親指の自然な動線」を必須要件として扱う。

UI実装・レビュー時は以下を必ず確認する。

- 主要操作は画面下部の親指が届きやすい範囲に配置する。
- 選択、確認、実行の流れが視線と指の移動に沿っている。
- 現在の状態、選択中の対象、次に実行される操作が一目で分かる。
- 頻繁に使う操作ほど下部・近距離に置く。
- 戻る、閉じる、削除、解除などの操作は誤タップしにくい配置にする。
- タップ対象は原則44px以上を確保する。
- 横スクロールや細かすぎるタップ操作は避ける。
- リスト、タブ、フィルタ、実行ボタンの動線が分断されていないか確認する。
- PWA/スマホブラウザでの連打、誤ズーム、スクロール干渉も考慮する。

## 5. Think Before Coding

実装前に以下を徹底する。

- 前提を明示する。
- 成功条件を明確にする。
- 不確実な点があれば質問する。
- 複数の解釈がある場合は、勝手に選ばず選択肢を提示する。
- よりシンプルな方法がある場合は明示する。
- 要求に対して過剰・危険・非効率な方針がある場合は、理由を添えて指摘する。

## 6. Simplicity First

最小限のコードで目的を達成する。

- 依頼されていない機能を追加しない。
- 単発用途の抽象化をしない。
- 不要な柔軟性や設定項目を追加しない。
- 起こり得ないケースへの過剰なエラーハンドリングをしない。
- 200行で書いたものが50行で済むなら、簡潔な形に書き直す。
- シニアエンジニア視点で過剰設計に見える場合は簡素化する。

## 7. Surgical Changes

変更は必要最小限にする。

- 関係ないコード、コメント、フォーマットを改善しない。
- 壊れていない箇所をリファクタリングしない。
- 既存のスタイルに合わせる。
- 無関係なデッドコードを見つけても、削除せず必要なら報告だけする。
- 自分の変更で不要になった import、変数、関数は削除する。
- 既存の未使用コードは、依頼されない限り削除しない。
- 変更したすべての行が、ユーザーの依頼に直接結びつく状態にする。

## 8. Goal-Driven Execution

作業を検証可能なゴールに分解し、確認まで行う。

- 実装前に成功条件を明確にする。
- バグ修正では、可能なら再現確認をしてから修正する。
- バリデーション追加では、無効入力の確認を含める。
- リファクタリングでは、変更前後で動作が維持されることを確認する。
- 複数ステップの作業では、簡潔な計画を示す。

計画の形式:

1. [作業内容] → verify: [確認方法]
2. [作業内容] → verify: [確認方法]
3. [作業内容] → verify: [確認方法]

成功条件が曖昧な場合は、作業前に確認する。

## 9. Sub Agent Review

レビュー専用サブエージェントは `.agents/` 配下の定義に従って、必要な場合のみ呼び出す。

- サブエージェントは実装担当ではなく、計画・差分・テスト結果・仕様案のレビュー担当とする。
- 呼び出す前に、対象範囲、前提、成功条件、確認してほしい観点を明示する。
- 呼び出し結果はそのまま採用せず、メインエージェントが妥当性を判断して最終方針に反映する。
- 複数サブエージェントを呼ぶ場合は、重複しない観点に分ける。
- 小規模・単一ファイル・既存テストで確認できる低リスク変更では、原則として呼び出さない。
- 指摘は `Blocking issues`, `Non-blocking issues`, `Missing verification`, `Verdict` の形式で整理する。

呼び出し目安:

- `qa-regression`: テスト、再現手順、回帰リスク、リリース前確認が必要な場合。
- `mobile-ui-ux`: UI、CSS、画面遷移、タップ操作、モバイル幅の表示に影響する場合。
- `game-logic`: 戦闘、探索、状態管理、マップ生成、装備、魔法、契約などのゲーム仕様に影響する場合。
- `balance-simulation`: 敵、報酬、ドロップ、成長、経済、難易度、進行速度に影響する場合。
- `content-design`: アイテム、敵、魔法、契約、説明文、表示テキストなどのコンテンツ追加・変更を行う場合。

以下の場合は、リスクに応じて該当サブエージェントのレビューを検討する:

- ユーザーが明示的にレビューを依頼した場合。
- 複数ファイルにまたがり、仕様・状態・UIフロー・ゲームバランスに影響する変更を行う場合。
- `src/data.js`, `src/data/*`, `src/rules/*`, `src/systems/*`,
  `src/state.js`, `src/state/*`, `src/combat_logic.js`,
  `src/combat_logic/*`, `src/combat.js`, `src/combat_ui/*`,
  `src/menu.js`, `src/menu/*`, `src/shop.js`, `src/shop/*`,
  `src/ui.js`, `src/ui/*`, `src/style.css`, `src/styles/*`,
  `tests/ui-ux.spec.js` に実質的な挙動変更が入る場合。
- UI変更でモバイル片手操作の品質が変わる可能性が高い場合。
- 数値変更でゲームバランスが変わる可能性が高い場合。

以下の場合は、サブエージェントレビューを省略してよい:

- 文言、コメント、軽微なスタイル、テスト期待値の明確な修正のみの場合。
- 単一ファイル内の局所的なバグ修正で、再現確認または既存テストにより検証できる場合。
- facade の import/export 調整など、挙動差分がないことを差分で確認できる場合。

レビュー後は、実装または回答に以下を含める:

1. 呼び出したサブエージェント
2. 採用した指摘
3. 採用しなかった指摘と理由
4. 実施した検証
