# 設計: 罠3択の再設計とラン毎生成への適合

日付: 2026-07-20
状態: 実装済み

## 背景

ソロ深度ローグライトへの転換（`2026-07-18-solo-depth-roguelite-design.md`）で
2つの前提が変わったが、罠システムは旧前提のまま残っている。

- マップがラン毎に再生成される（`runSeed` ベース）
- フロア間が不可逆（引き返し不可）、パーティは1人

結果として3つの破綻が起きている。

### 1. 罠の永続化が機能していない

`trapPersistenceByDepth` と `persistDungeonTraps` は固定マップ前提の設計。

- `trapId` は `trap_${floor}_${x}_${y}` の座標ベース
- マップはラン毎に再生成されるため、次ランで同じ座標に罠が来る保証がない
- `applyDungeonMemoryToMaps` は `save_storage` 経由でしか呼ばれず、
  ラン中のフロア生成経路（`ensureRunFloor`）は memory を一切適用しない

「罠は弱体化して残るかもしれない」「罠を完全に破壊した！」というログは
プレイヤーへの虚偽表示になっている。

### 2. 迂回のコストが罠難度と無相関

`calculateSuccessRate` は `- trap.difficulty - (floor-1)*5` で深度とともに落ち、
深層では下限10%に張り付く。一方で迂回は常に「歩数+5・エンカウント25%」の固定コスト。
深いほど迂回が相対的に強くなり、深層は「全部迂回」が支配戦略になる。

### 3. 強行突破が死に選択肢

通常罠の強行は判定なしのフルダメージ確定で、歩数ゼロが唯一の対価。
解除の期待値を下回るため選ばれない。落とし穴だけ跳躍判定を持つ非対称も残っている。

## 決定サマリ

| 論点 | 決定 |
| --- | --- |
| スコープ | 3択の再設計＋永続化の撤去 |
| 罠の役割 | ルート選択の障害物 |
| 事前可視化 | 隣接マスでの自動察知 |
| 迂回コマンド | 削除し、マップ移動に還元 |
| 罠の配置 | 深度でチョークポイント率を上げる |
| 強行のコスト | 確定・軽減ダメージ |
| 解除成功率 | クラス適性で二極化 |
| セーブ互換 | 考慮しない |

## 1. 削除範囲

### 撤去する

- `src/systems/traps.js` の `trapPersistenceByDepth`
  （`keepWeakenedRate` / `reactivateRate` / `permanentDisarmCount`）
- `src/result.js` の `persistDungeonTraps` 全体と、`triggerRunResult` からの呼び出し
- `src/state/dungeon_state.js` の `applyDungeonMemoryToMaps` のうち traps ループのみ
- `src/systems/traps.js` の `addDisarmPersistenceLog`
- `trap.weakenLevel`、`trap.state === "weakened"`、`weakenedModifiers`、
  `calculateSuccessRate` の `weakenedBonus`
- `handleTrapAction` の `"bypass"` ブランチと対応するUIボタン

`weakened` の撤去は `triggerTrap` / `triggerPitfall` / `alarmWeakened` の
引数まで波及する。本設計で最大の差分。

### 残す

- `state.dungeonMemory` 本体。`mapFragments` を `src/renderer.js` が使用中のため、
  `traps` キーへの書き込みだけを止める
- `trap.state` の `hidden` / `discovered` / `disabled` の3値
- `back`（引き返す）。チョーク罠で別方向を探る導線として必要
- `trapsDisarmed` / `trapsTriggered` / コデックス記録。ラン単位の統計として存続

### セーブ互換

旧セーブ破棄が既定方針。`dungeonMemory.traps` は読み捨て、マイグレーション処理なし。

## 2. 生成 — チョーク率の制御

### チョーク判定

候補セルを1つ塞いだ状態でスタートから BFS し、下り階段へ到達不能なら
チョークポイントと判定する。

```
isChokeCell(grid, cell, start, stairsDown):
  cell を通行不能にして start から到達可能セルを列挙
  stairsDown が含まれなければ true
```

Tarjan の関節点列挙は採らない。関節点は「グラフ全体を切る点」であって
「スタートと階段を切る点」ではなく、判定を合わせる補正が別途必要になる。
マップは30×30で歩行可能セルは数百なので、候補ごとの BFS 総当たり
（概算21万辺走査、フロア生成1回きり）で十分速く、
既存の `getReachableCellKeys` をそのまま再利用できる。

### 配置アルゴリズム

1. 既存の `trapCandidates` 収集はそのまま
2. 各候補を `isChokeCell` で choke / non-choke に二分
3. 深度からチョーク目標数を算出し、choke プールから抽選、残りを non-choke から抽選

### チョーク率

深度は無限スケールのため、B5でカンストする既存の `getDepthCategory` は
罠配置には使わず、連続式を用いる。

```
chokeRate = clamp(0.10 + 0.04 * (floor - 1), 0.10, 0.55)
```

B1=10% / B5=26% / B10=46% / B12以降=55%。
上限55%は必須。全てを関所にすると回避判断が消える。

### 罠数の上限

現行の `6 + floor` は上限がなく、B30で36個になる。
`min(6 + floor, 16)` に変更する。深層は数ではなくチョーク率と難易度で締める。

### 詰み保証

チョーク罠は階段への唯一経路上に立つ。強行突破が必ず成功することが、
フロア突破不能を防ぐ唯一の保証になる。強行に判定を入れない設計判断はここに紐づく。
HPが尽きて死ぬことはあるが、それは詰みではなくラン終了。

### フォールバック

choke プールが目標数に満たない場合（迷路形状によっては関節点が少ない）、
あるだけ使って残りを non-choke で埋める。目標に届かなかった事実は
生成メタに記録し、テストで分布を検証できるようにする。

`scratch/sim_trap_choke.js` を各深度100フロアで実行した結果:

| 深度 | target | actual | shortfall |
| --- | ---: | ---: | ---: |
| B1 | 0.100 | 0.141 | 1% |
| B3 | 0.180 | 0.143 | 0% |
| B5 | 0.260 | 0.286 | 0% |
| B8 | 0.380 | 0.370 | 4% |
| B10 | 0.460 | 0.489 | 9% |
| B12 | 0.550 | 0.562 | 14% |
| B15 | 0.550 | 0.563 | 11% |
| B20 | 0.550 | 0.505 | 17% |

本番のラン内生成経路では全深度でチョーク配置が機能し、実測率は目標率に近い。
候補不足によるフォールバックも最大17%で、調整基準の5割を下回った。

## 3. 察知と表示

### 隣接察知に一本化

現行の「踏んだ瞬間の判定」（`handleTrapStepCheck`）を廃止し、
`traceRead` 保持者限定だった隣接察知を全キャラの基本能力に格上げする。

踏んでからの判定が残ると、隣接察知に失敗しても踏めばもう一度引ける
二重構造になり、事前情報の価値が失われる。

### 判定タイミング

移動確定のたび、隣接4マスの `hidden` 罠に対して1回だけ判定する。

- 壁越しは察知しない。隣接セルでも間に壁があれば対象外
  （`canEnterFrom` 相当のチェックを噛ませる）。
  行けない場所の情報でマップが汚れるのを防ぐ
- 1つの罠につき判定は生涯1回。`trap.detectRolled` フラグで管理。
  引き直せると判定が作業に化ける

### 察知率

クラスボーナスなし。全クラス共通。

```
察知率 = 0.85 - 0.015 * (floor - 1)   下限 0.60
```

罠の役割が「ルート選択の障害物」である以上、情報は全員に配らないと
選択が成立しない。見えない罠は選択ではなく事故であり、
事故枠は残り15〜40%で足りる。

察知と解除の両方にクラス適性を掛けると、非適性クラスが
「見えない」＋「解除できない」で二重に罰され、
残る答えが耐久を積むことだけになる。これを避けるため軸を分離する。

- 察知（情報）: 全クラス共通
- 解除（処理）: クラス適性で差をつける

非適性クラスの答えは「回り込む」か「強行で確定小ダメージを払う」になる。
浅層は回り込めるのでほぼ無傷、深層はチョークが増えて強行の確定ダメージが効く。
クラス差が深度とともに開き、浅層で非適性クラスが理不尽に殴られることもない。

### traceRead アフィックス

察知率の底上げではなく、`revealLevel` を上げて罠種別まで見せる役割に寄せる。
察知率が全員高い前提と衝突させない。

### 表示

`discovered` 罠を2Dマップと3D通路ビューの両方にマーカー表示する。
宝箱アイコンの3Dビュー表示（コミット `1b44050`）と同じ経路を踏襲する。
マーカーの情報量は既存の `revealLevel` 段階を流用し、
「罠がある」→「種別が分かる」の2段階とする。

### 踏む前に引き返せること

`discovered` 罠のマスへ移動しようとした時点で `trap_encounter` に入る。
現行は移動後に発生するが、ルート選択として機能させるには
「そのマスに入る直前」に判断を挟む必要がある。
`back` は移動前の位置に留まるだけとなり、歩数コストもゼロになる。

## 4. 3択の数値設計

### 通常罠: 解除 / 強行 / 引き返す

| 選択 | 判定 | 結果 |
| --- | --- | --- |
| 解除 | `roll < rate` | 無傷通過 |
| 解除 | `rate ≤ roll < rate + 15` | 軽減ダメージ、通過 |
| 解除 | `roll ≥ rate + 15` | フルダメージ、通過 |
| 強行 | なし | 軽減ダメージ確定、通過 |
| 引き返す | なし | コストゼロ、その場に留まる |

引き返す以外の4分岐はいずれも `trap.state = "disabled"` にして通過する。
現行の部分成功だけ `discovered` に戻す挙動は廃止する。
罠は解除されたか発動したかのどちらかであり、
同じ罠を再度踏んで判定を引き直せる状態を残さない。

軽減倍率は 0.5。旧 `weakenedModifiers.effectPower` の値を
`FORCE_DAMAGE_MULTIPLIER` として残す。

解除は「無傷 or フル」の博打、強行は「確定で半分」。
期待値は解除がやや上、分散は強行がゼロ。
HPが十分なら解除、削れていれば強行、という読み替えが成立する。

### 解除成功率

現行の `luk + agi` 生加算を廃止する。
これが浅層95%・深層10%への張り付きの原因であり、
ステータス2つを直接足すためスケールが制御できない。

```
適性 (Thief/Ninja/Ranger): rate = 80 + lv * 1.0 - (floor - 1) * 2.0   clamp 20..90
非適性:                    rate = 40 + lv * 0.5 - (floor - 1) * 2.0   clamp 5..60
解除系アフィックス: 加算補正
```

アフィックスの補正値は既存定義（`src/data/affixes.js`）に合わせるものとし、
本設計では加算方式であることのみを決める。
乗算補正は clamp と噛み合わず上限に張り付くため採らない。

- 適性: B1で81% / B10Lv10で72% / B20Lv20で62%。深層でも主軸として機能
- 非適性: B1で40% / B10Lv10で27% / B20Lv20で12%。
  浅層は安いギャンブル、深層は強行と回り込みに寄る

毎ランLv1開始のため `lv` はラン内の伸びをそのまま反映する。

`trap.difficulty` の乱数（±5）は式から外す。
チョーク率と深度で既に締めており、
表示成功率に見えないブレを足すと判断材料として濁る。

### 落とし穴: 縁を伝う / 飛び込む / 引き返す

落とし穴は次フロアへ強制降下する。片道潜行かつ深度がスコアである以上、
これはペナルティではなくショートカットであり、
ダメージと引き換えに残りのフロア探索を捨てて1階深く進む取引になる。
軸が異なるため選択肢の文言を変える。

| 選択 | 判定 | 結果 |
| --- | --- | --- |
| 縁を伝う | `roll < rate + 20` | フロアに留まって通過 |
| 縁を伝う | 失敗 | 落下（ダメージ＋次フロアへ） |
| 飛び込む | なし | 軽減ダメージ＋次フロアへ確定 |
| 引き返す | なし | その場に留まる |

現行の「飛び越える」（`rate - 20` の判定で向こう岸へ渡る）は廃止する。
フロアを飛ばせる取引のほうが、片道潜行と深度スコアという骨格に噛み合う。
落とし穴がチョークに置かれた場合も、飛び込みが確定成功のため詰まない。

## 5. モジュール構成

`src/systems/traps.js` は察知・成功率・3択処理・効果発動・永続化を
1ファイルで抱えている。永続化を抜いても生成側のチョーク判定が加わるため、
責務を分割する。

- `src/systems/traps.js` — 罠エンカウントの状態遷移と3択処理。UIからの入口
- `src/rules/trap_rules.js`（新規）— 純関数のみ。
  `calculateDisarmRate(classId, level, floor, affixes)`、
  `calculateDetectRate(floor)`、`getForceDamageMultiplier()`
- `src/map_generator.js` — `isChokeCell` と配置抽選。既存の到達判定ヘルパの隣

数値式を `trap_rules.js` に隔離するのが要点。
現行の `calculateSuccessRate` は `state.party` を直接読むため、
率の検証にゲーム状態の組み立てが必要になっている。
引数だけで決まる純関数にすればシミュレータからも直接叩ける。

## 6. テスト

- `scratch/test_traps.js` 拡張 — 3択それぞれの結果分岐、
  `weakened` 撤去後の状態遷移、察知の1回限り保証、壁越し非察知
- `scratch/test_trap_rules.js`（新規）— 率式の境界値。
  適性/非適性 × 深度 × レベルの表を固定値でアサート。clamp の上下端
- `scratch/sim_trap_choke.js`（新規）— B1〜B20を各100フロア生成し、
  実チョーク率が目標式に収束するか、フォールバック発動頻度、
  罠数上限を検証。`sim_maze_metrics.js` の書き方を踏襲
- `tests/verify-chest-trap-ui.spec.js` の隣に3択UIのブラウザ検証を追加。
  迂回ボタンの消滅、落とし穴のラベル差し替えを確認

## 7. 実装順

各段でプレイアブルを維持する。

1. 永続化撤去と `weakened` 削除。差分が最大なので先に土台を平らにする
2. `trap_rules.js` 切り出しと率式の置換
3. 隣接察知への一本化＋マーカー表示
4. 生成側のチョーク配置
5. 3択の再定義（迂回削除・強行の軽減確定・落とし穴の飛び込み）

1と2は挙動が変わらないリファクタに近く、3以降で体験が変わる。
